# ADR-0006: Progressive Web App (PWA) as the Frontend

**Date**: 2026-04-18  
**Status**: Accepted

---

## Context

Elixir targets users who want to track personal finances on both mobile (quick expense logging, OTP-based auth, checking balances on the go) and desktop (uploading statements, reviewing analytics, setting budgets). The frontend delivery mechanism must work well on both form factors.

Options:
1. Native iOS app + native Android app (separate codebases or React Native)
2. Progressive Web App (single codebase, installable on both platforms)
3. Responsive web app only (no installability, no offline)

---

## Decision

Build the frontend as a **Progressive Web App (PWA)**: a responsive web application with a Web App Manifest and Service Worker that enables installation on mobile home screens and desktops.

---

## Consequences

### Positive

- **Single codebase for all platforms.** One React (or Vue) app targets mobile, tablet, and desktop browsers. No platform-specific code for UI components.
- **No App Store.** Users install by visiting the URL and tapping "Add to Home Screen". No App Store review, no update delays, no store submission process. Critical fixes can be deployed immediately.
- **Immediate updates.** When a new version is deployed, users get it on their next visit. No "update available" prompt that users dismiss for weeks.
- **Works in the browser too.** Users who don't want to install it can use it as a regular web app on any browser.
- **Offline capability (future).** Service Workers can cache recently viewed data for offline viewing. Not implemented initially but architecturally available.
- **Cost.** No Apple Developer Account ($99/year), no Google Play Developer account ($25 one-time). No App Store revenue share on in-app purchases (not applicable here, but worth noting).

### Negative / Trade-offs

- **iOS PWA limitations.** Apple's PWA support has historically been limited:
  - iOS does not support Web Push Notifications for PWAs in all regions (improving post-iOS 16.4)
  - Some iOS PWA behaviors differ from Android (e.g., splash screen handling, status bar theming)
  - The "Add to Home Screen" flow on iOS is less discoverable than the Android install prompt
  - This is the most significant trade-off. Mitigated by: the app functions identically in Safari without installation; PWA support on iOS continues to improve with each release.
- **No native device API access.** Cannot use iOS Health Kit, Face ID (via native SDK), or camera for receipt scanning without bridging via web APIs. WebAuthn (for Face ID/Touch ID as a future auth layer) is a web standard and is available.
- **No App Store discoverability.** Users must find the app via its URL. For a personal productivity tool this is acceptable — users are invited/self-directed, not discovered via app store search.

---

## Frontend Tech Stack (to be finalised before implementation)

The PWA frontend is not yet implemented. When starting, the recommended choices are:

| Concern | Recommendation | Why |
|---|---|---|
| Framework | **React** | Largest ecosystem, most PWA tooling, team familiarity |
| Build tool | **Vite** | Fast HMR, native PWA plugin (`vite-plugin-pwa`) |
| PWA plugin | **vite-plugin-pwa** | Workbox-based Service Worker, manifest generation |
| Charting | **Recharts** or **Chart.js** | React-friendly, adequate for financial charts |
| State management | **Zustand** or **React Query** | Lightweight; React Query handles server state natively |

This decision will be made when frontend implementation begins. The backend API (FastAPI with OpenAPI docs) is frontend-framework-agnostic.

---

## Alternatives Considered

**Native iOS + Android (separate)**: Highest quality native experience, full device API access. Requires either two separate codebases (Swift + Kotlin) or React Native. At the current team size, maintaining two codebases is impractical. React Native is a viable option if native performance becomes critical but adds complexity over PWA without strong justification at this stage.

**React Native**: Single codebase for iOS and Android. Better native API access than PWA. Requires App Store submission (iOS). Adds Expo or bare React Native as an additional toolchain to maintain. The benefits do not outweigh the complexity for a financial tracking tool where the primary interactions are form fills, file uploads, and chart views — none of which require native performance.

**Responsive web app (no PWA features)**: Simpler — no Service Worker, no manifest. Ruled out because installability significantly improves the mobile experience (home screen icon, full-screen mode, no browser chrome). The additional complexity of adding a manifest and Service Worker is small.
